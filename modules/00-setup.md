## Key Zig Concepts

### `build.zig` — The Build System

Zig ships its own build system. Instead of a `Makefile` or `CMakeLists.txt`, you write `build.zig` — a normal Zig program that calls into the `std.Build` API. This gives you the full expressiveness of the language at build time.

```zig
// Every build script exposes this entry point.
pub fn build(b: *std.Build) void { ... }
```

The `b.standardTargetOptions` and `b.standardOptimizeOption` helpers let the user pass `-Dtarget=` and `-Doptimize=` flags without you writing any argument parsing yourself.

In Zig 0.15, executables are created by first creating a **Module** and then passing it as `root_module`:

```zig
const exe = b.addExecutable(.{
    .name = "redis-zig",
    .root_module = b.createModule(.{
        .root_source_file = b.path("src/main.zig"),
        .target = target,
        .optimize = optimize,
    }),
});
```

### `build.zig.zon` — Package Manifest

`build.zig.zon` is a Zig Object Notation file (`.zon`) that declares the package name, version, and dependencies. Think of it as `package.json` or `Cargo.toml`.

### `std.net` — Networking Primitives

Zig's standard library provides low-level, allocation-free networking in `std.net`. The key types are:

| Type | Purpose |
|---|---|
| `std.net.Address` | Represents an IP+port endpoint |
| `std.net.Server` | A bound, listening TCP socket |
| `std.net.Server.Connection` | An accepted client connection |

### Error Handling with `!` and `try`

Zig uses **error unions** instead of exceptions. A function that can fail returns `!ReturnType`. The `try` keyword propagates errors up the call stack — it is syntactic sugar for:

```zig
const result = someFunction() catch |err| return err;
```

### `defer` — Guaranteed Cleanup

`defer` schedules an expression to run when the enclosing scope exits, regardless of whether it exits normally or via an error. This is how Zig handles RAII:

```zig
var server = try address.listen(.{});
defer server.deinit(); // always runs, even on error
```

## Task 3: Write `build.zig`

```zig
// build.zig
const std = @import("std");

pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    const exe = b.addExecutable(.{
        .name = "redis-zig",
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });

    b.installArtifact(exe);

    const run_cmd = b.addRunArtifact(exe);
    run_cmd.step.dependOn(b.getInstallStep());
    if (b.args) |args| run_cmd.addArgs(args);

    const run_step = b.step("run", "Run the Redis server");
    run_step.dependOn(&run_cmd.step);

    const test_exe = b.addTest(.{
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });

    const run_tests = b.addRunArtifact(test_exe);
    const test_step = b.step("test", "Run unit tests");
    test_step.dependOn(&run_tests.step);
}
```

Key points:
- `b.createModule` creates a compilation unit from a source file.
- `b.installArtifact` copies the binary to `zig-out/bin/` on `zig build`.
- The `"run"` step lets you do `zig build run` to launch the server directly.
- The `"test"` step runs all `test` blocks defined in `src/main.zig`.

## Task 4: Write the TCP Server in `src/main.zig`

This is the full Module 0 server. It accepts exactly one connection, prints the client's address, then exits.

```zig
// src/main.zig
const std = @import("std");

pub fn main() !void {
    // Parse "127.0.0.1:6379" into a std.net.Address.
    const address = try std.net.Address.parseIp4("127.0.0.1", 6379);

    // Bind and start listening. reuse_address = true avoids "address already
    // in use" errors during rapid restarts (the SO_REUSEADDR socket option).
    var server = try address.listen(.{
        .reuse_address = true,
    });
    defer server.deinit();

    std.debug.print("Listening on 127.0.0.1:6379\n", .{});

    // Block until one client connects.
    const connection = try server.accept();
    defer connection.stream.close();

    std.debug.print("Client connected from {any}\n", .{connection.address});
    std.debug.print("Closing connection.\n", .{});
}
```

### What happens step by step

1. **`std.net.Address.parseIp4`** — Parses the dotted-decimal string `"127.0.0.1"` and port `6379` into a typed `Address` struct. The `!` means it can fail (e.g. if the string is malformed); `try` propagates any error to `main`'s caller (the OS).
2. **`address.listen`** — Creates a TCP socket, calls `bind()` and `listen()` on it, and returns a `std.net.Server`. The `defer server.deinit()` ensures the socket is closed when `main` returns, even on error.
3. **`server.accept()`** — Blocks (calls the OS `accept()` syscall) until a client connects. Returns a `Connection` containing a `stream` (bidirectional `std.net.Stream`) and the client's `address`.
4. **`connection.stream.close()`** — Closes the accepted socket. The `defer` guarantees it runs.

## Task 5: Build and Run

```bash
zig build
zig build run
```


```
Listening on 127.0.0.1:6379
Client connected from 127.0.0.1:XXXXX
Closing connection.
```

## Under the Hood: The Build Cache

When you run `zig build`, Zig creates two directories:

- `.zig-cache/` — per-project incremental build artifacts (safe to `.gitignore`)
- `~/.cache/zig/` — global cache shared across all projects

The global cache is why cold builds are slow but subsequent builds are fast. Zig uses a content-addressed cache: if the source and all compiler inputs hash to the same value, the cached artifact is reused.

> **Tip:** Add `.zig-cache/` and `zig-out/` to your `.gitignore`.
